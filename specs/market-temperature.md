## 1. 项目目标

每个交易日收盘后,自动抓取 A 股基础市场数据,计算一个 0–100 的「市场温度评分」,在网页上展示当日评分、分项数据和历史走势。

**要解决的问题**:雪球舆情这类数据太虚、没有操作参考性。本工具改用客观的市场统计数据,每天给出一个直观、可纵向对比的分数。

## 2. 与现有仓库的关系(新增)

本功能不是独立项目,而是「八类资金监控」仓库的一个新模块,复用其全部基础设施:

| 现有设施 | 复用方式 |
|---|---|
| `src/utils.py` | `is_trading_day` 交易日历、`safe_fetch`/`cached_fetch` 重试+缓存、`upsert_history` 落盘 |
| `src/collectors/` 模式 | 新增 `src/collectors/temperature.py` 采集,评分逻辑放 `src/temperature_score.py`(或并入 analyzer) |
| `src/collectors/market_common.py` | 两市成交额(交易所官方接口)、指数涨跌幅(中证官网)**已有现成实现,直接调用** |
| `--mock` 离线模式 | 新模块必须接入 `src/fixtures.py`,保证 `python src/main.py --mock` 全流程可测 |

## 3. 数据指标(v1)

每个指标标注:**方向**(↑ = 值越大温度越高;↓ = 值越大温度越低)与数据源(主 → 备)。

### 市场宽度(权重 35%,组内等权)

| 指标 | 方向 | 数据源 |
|---|---|---|
| 上涨家数占比(上涨 ÷ 涨跌平总数) | ↑ | 乐咕 `stock_market_activity_legu`(**唯一源,已实测可用**)。原计划的主源 `stock_zh_a_spot_em` 和备源 `stock_zh_a_spot`(新浪)**实测均挂起超时(~27s)不可用**,不再作为备源;乐咕失败则该指标当日缺席 |
| 涨停家数(剔除 ST 与上市首日新股) | ↑ | 东财涨停池 `stock_zt_pool_em`(主,**已实测可用**,33 只/16 列)→ 乐咕 activity 的涨停数(备,口径含 ST,需标注) |
| 跌停家数 | **↓** | 同上(东财跌停池 `stock_zt_pool_dtgc_em`,主,**已实测可用**,192 只/16 列 → 乐咕) |
| 连板家数、最高连板高度 | ↑ | 东财涨停池「连板数」字段(**已实测可用**,字段名待实现阶段用 `df.columns` 核对);**无独立备源,当日接口失败则缺席** |

> 上涨/下跌用「占比」而非绝对家数,避免上市公司总数逐年增长污染 250 日分位数。

### 情绪强度(权重 30%,组内等权)

| 指标 | 方向 | 数据源 |
|---|---|---|
| 涨停封板率 = 收盘涨停 ÷(收盘涨停 + 炸板) | ↑ | 东财炸板池 `stock_zt_pool_zbgc_em`(**已实测可用**,10 只/16 列);**无独立备源,当日接口失败则缺席** |
| 昨日涨停股今日平均涨幅 | ↑ | 东财昨日涨停池 `stock_zt_pool_previous_em`(**已实测可用**,43 只/16 列);**无独立备源,当日接口失败则缺席** |

- 封板率分母为 0(当日无涨停且无炸板的极端冰点日)时,该指标记 0 分位(冰点),不做除法。

### 量能(权重 20%,组内等权)

| 指标 | 方向 | 数据源 |
|---|---|---|
| 两市总成交额 | ↑ | `market_common.sse_stock_turnover` + `szse_stock_turnover`(交易所官方,生产已用)。⚠️ 2026-07-19 回归探测中**两者都失败**——`stock_sse_deal_daily` 返回结构不匹配(`ValueError: Length mismatch`,像是接口返回格式变动,重试无用)、`stock_szse_summary` 连接被重置(`ConnectionResetError`,像是偶发抖动)。现有 `market_common.py` 已对空/异常结构做了防御并返回 `None`,不会导致流程崩溃,但**不能假设这两个接口当天必然成功**,量能组要按 §10 兜底走 |
| 当日成交额 ÷ 近 5 日均额 | ↑ | 由上一指标的历史序列现算 |

### 指数(权重 15%,组内等权)

| 指标 | 方向 | 数据源 |
|---|---|---|
| 上证综指当日涨跌幅 | ↑ | `market_common.csindex_day("000001", ...)`(中证官网,**已实测可用**——此前担心 SSE 自编指数不被中证指数网收录,实测证明收录) |
| 深证成指当日涨跌幅 | ↑ | 中证官网**实测不覆盖**(`symbol="399001"` 返回空数据)。改用新浪 `stock_zh_index_daily(symbol="sz399001")`(**已实测连通**,用 `sh000001` 验证过连通性,399001 待补测;无「涨跌幅」列,需用收盘价环比自算)→ 东财 `stock_zh_index_daily_em`(备,同样已实测连通) |
| 创业板指当日涨跌幅 | ↑ | 同深证成指,中证官网不覆盖(`symbol="399006"` 同样返回空数据),改用新浪/东财备源(`sz399006`) |

## 4. 评分模型

1. **分位数化**:每个指标取其近 250 个交易日历史序列,算当日值的分位数(0–100)。**反向指标(↓)取 `100 − 分位数`**。
2. **组内合成**:组内各指标等权平均(v1 简化;调参放到回测阶段)。
3. **加权总分**:宽度 35% + 情绪 30% + 量能 20% + 指数 15%。
4. **档位**:0–20 冰点 ｜ 20–40 偏冷 ｜ 40–60 中性 ｜ 60–80 偏热 ｜ 80–100 过热。

### 冷启动与历史回填(新增)

- **能回填的先回填**:两市成交额、指数涨跌幅、**以及涨停池系列(实测已确认可用)**都支持按 `date` 参数取历史数据,可一次性回填 250 个交易日。回填脚本一次性运行,结果直接写入 `history.json`。乐咕 `stock_market_activity_legu` 按接口设计是当前快照服务、不带 `date` 参数(此次探测未专门验证这一点,只验证了连通性),大概率回填不了,上涨家数占比只能从上线日开始积累(见下一条);实现阶段应先确认一下。
- **回填不了的窗口自适应**:上涨家数占比等仅有当日快照的指标,从上线日开始积累;其分位数窗口 = min(已积累天数, 250),**积累不足 60 个交易日期间,页面在该指标卡片上标注「样本不足」**。
- **总分门槛**:上线首日起即出总分(靠可回填指标撑起),但前 20 个交易日页面在大分数旁标注「建仓期,分数参考性有限」。

### 回测前提(新增,硬要求)

每日落盘**必须包含每个指标的原始值**(raw),而不仅是分位数和总分。否则一个月后回测调权重时无法用新权重重算历史。见 §6 数据格式。

**结论与调整**(已同步到 §3 表格):

1. **涨停池系列全部可用**,比 v1.1 假设的「东财大概率不通」乐观得多——push2ex 主机在 Actions 出口 IP 下与 82.push2/48.push2(全市场快照走的主机)表现不同。情绪强度组**不需要**降级方案,权重维持原设计;v1.1 里"若涨停池不可用则整组缺席归一化"这条预案**保留作为运行时兜底**,但不再是预期路径。
2. **上涨家数占比反而是最脆弱的一环**:计划的主源和备源都挂起超时(而非快速拒绝——两次都卡到 27s 左右才失败,是连上了但无响应,不是连接被拒)。改用乐咕作唯一源;这也是市场宽度组里目前唯一没有真正备源的指标,乐咕失败当天就缺这一项。
3. **「生产已验证」不等于「稳定」**:两市成交额的两个接口这次回归探测都失败了,且是两种不同的失败模式(结构不匹配 / 连接重置),说明即使是本仓库其他模块每天在用、目前看起来跑得好好的接口,也会偶发失败。`market_common.py` 现有的空值防御能接住,不会导致整体流程崩溃,但印证了 §10 兜底策略(缺席 + 权重归一化)是刚需,不是防御性过度设计。
4. **指数组有一半好消息一半坏消息**:上证综指可以走中证指数网(和沪深300一样稳);深证成指、创业板指必须走新浪/东财备源,且这两个备源接口都不带现成的「涨跌幅」列,采集器要自己拿收盘价环比算(指数本身不存在除权问题,直接算就行)。
5. 所有采集仍然走现有 `safe_fetch`(重试 + 超时)框架,不裸调 AkShare。挂起超时(而非快速失败)的接口(如全市场快照)应参考 `leverage.py` 里的做法调低 `retries`/`hard_timeout`,避免拖慢整个 workflow。

## 6. 存储格式

单一数据源 `docs/temperature_data.json`(随 Pages 发布,前端只读这一个文件;同时它就是历史存档,不再另设每日单文件):

```json
{
  "updated": "2026-07-21",
  "days": [
    {
      "date": "2026-07-21",
      "score": 68.3,
      "band": "偏热",
      "groups": {"breadth": 72.1, "sentiment": 65.0, "volume": 70.4, "index": 58.2},
      "raw": {
        "adv_ratio": 0.63, "limit_up": 45, "limit_down": 3,
        "max_board": 5, "boards_ge2": 12,
        "seal_rate": 0.78, "yesterday_zt_ret": 2.1,
        "turnover_yuan": 1.85e12, "turnover_vs_5d": 1.12,
        "sh_chg": 0.8, "sz_chg": 1.1, "cyb_chg": 1.6
      },
      "pct": {"adv_ratio": 71.0, "limit_up": 80.4, "...": "各指标当日分位数"},
      "missing": [],
      "low_sample": [],
      "warm_up": false
    }
  ]
}
```

- `missing`:当日缺席的指标名,前端据此展示「暂缺」。
- `low_sample`:该日样本数 < 60(`config.yaml` 的 `low_sample_threshold`)的指标名列表,对应 §4「样本不足」标注。回填过的指标(成交额、指数、涨停池系列)上线首日就有 250 天样本,不会出现在这里;只有 `adv_ratio`(乐咕无历史查询)会在上线后前 60 个交易日出现。
- `warm_up`:该日是否在上线后前 `warm_up_days`(默认 20)个交易日内,对应「建仓期,分数参考性有限」提示。
- `score`/`band` 为 `null`:当日可用分组权重之和低于 `min_score_weight`(默认 0.5)时不出总分,由前端展示「今日数据缺失」,历史曲线在此处留断点(不连接前后两天)。
- 权重、组内成员、反向指标、窗口/阈值、档位全部写在 `config.yaml` 的 `temperature:` 段,不硬编码,方便回测后调整——调权重/窗口后重跑 `webpage.write_temperature_data()` 即可用同一份 `data/temperature.csv` 原始值重新出全部历史,不需要重新抓数据。

## 7. 调度与架构

```
现有 daily.yml(北京时间 18:30,UTC 10:30,周一至周五)
  └─ python src/main.py(现有八类资金流程)
       └─ 末尾追加:温度采集 → 评分 → 写 docs/temperature_data.json
            └─ 现有「提交结果」步骤一并 commit(git add 已覆盖 docs/)
                 └─ 现有 deploy-pages job 一并发布
```

**不新开 15:40 的 workflow**,理由:

1. Actions cron 实际延迟常达 15–60 分钟,15:40 的名义时间没有意义;
2. 乐咕、交易所官方概况等盘后数据源需要时间更新,18:30 更稳(现有项目选 18:30 即是实践结论);
3. 两个 workflow 各自 commit 会产生 push 竞争,现有 `concurrency: daily-report` 组不跨 workflow 生效。

温度模块失败**不得**导致八类资金日报整体失败:在 main.py 中以独立 try/except 包裹,失败时日志告警、跳过温度部分。

## 8. 页面设计

`docs/temperature.html`,移动端优先(微信内打开):

- **顶部**:日期 + 大号评分(例:68 · 偏热)+ 较上一交易日变化箭头 + 0–100 温度计(meter,标 5 档边界)
- **中部**:分项卡片——涨跌家数、涨停/跌停、连板、封板率与打板情绪、成交额及放缩量、三大指数;缺席指标显示「暂缺」而非 0,样本不足的指标额外标「样本不足」
- **底部**:近 60 日评分折线(可切 250 日/全部),叠加 5 日均线(纯前端计算)以平滑单日抖动;不出分的日子在线上留断点(不连接前后两天),5 日均线只要有 5 个真实评分就继续算,不随断点中断
- **实现方式(非 vendor ECharts)**:`docs/index.html`、`docs/risk.html` 现有约定是纯手写内联 SVG + CSS 变量做明暗主题,零外部依赖;`temperature.html` 照此约定实现,不引入 ECharts 或任何前端库。且它是**静态文件**,不像 `index.html` 那样每天由 Python 重新生成——只有 `docs/temperature_data.json` 每天更新,页面用 `fetch()` 读取(与 `risk.html` 读 `leverage_data.json` 的模式一致)
- `docs/index.html` 首页已加入口链接

## 10. 边界与免责

- 节假日、非交易日由现有 `is_trading_day` 统一跳过。
- **兜底策略**:单指标失败→缺席并权重归一化,`missing` 字段记录;当日大面积失败→不出分,历史曲线留断点,页面显示「今日数据缺失」。**不用前一日数据冒充当日**。
- 涨停口径:剔除 ST 与上市首日无涨跌幅限制的新股;含北交所(东财池内 30% 涨跌幅个股),降级到乐咕口径时含 ST 并在页面标注。
- 评分只反映市场情绪温度,**不构成任何买卖建议**。
